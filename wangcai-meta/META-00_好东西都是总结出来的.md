# META-00: 好东西都是总结出来的 — 旺财知识增殖方法论

> 借鉴 shiji-kb META-00 哲学，适配旺财的个人AI助手场景。

## 核心信念

**知识不是设计出来的，是总结出来的。**

从实践中涌现的模式，比预先设计的方案更可靠、更持久。

## 四层知识升华路径

```
冷启动          模式涌现         知识固化         系统自治
┌──────┐      ┌──────┐      ┌──────┐      ┌──────┐
│ 人工  │ ──→ │ 边做  │ ──→ │ Skill │ ──→ │ 自主  │
│ 执行  │      │ 边总结 │      │ 封装  │      │ 运行  │
└──────┘      └──────┘      └──────┘      └──────┘
   ↓            ↓            ↓            ↓
 人工100%     人工60%      人工20%      人工<10%
```

### 什么时候该封Skill？

如果以下条件满足 ≥2 个，就应该做：

1. **重复发生** — 这个问题我处理过≥2次
2. **模式明确** — 解决方案有可重复的步骤
3. **有判断空间** — 不是简单的"固定答案"，需要根据情况变通
4. **老板问过 ≥1次** — 说明对他有价值

### 什么时候不该封Skill？

1. 一次性事件（如"帮我看一下这家公司"）
2. 纯数据查询（有API直接调）
3. 老板只是随口问问

## OTF + JIT 工作法

| 原则 | 含义 | 在旺财场景中的应用 |
|------|------|-------------------|
| **OTF** (On The Fly) | 边做边总结，不等完美 | 这次做日报遇到的坑 -> 立刻记到反思日志 |
| **JIT** (Just In Time) | 需要什么才学什么，不预加载 | 老板问A股时再分析，不提前备着 |
| **Bootstrap** | 已有的Skill帮助生成新Skill | book-to-skill 帮生成了 dewey-how-we-think |

## 知识资产复用金字塔

```
第四层: META（元技能） ← 如何设计Skill本身
第三层: Skill（技能）  ← 可复用的方法论封装
第二层: 模式（经验）   ← 从实践中提炼的规律
第一层: 数据（事实）   ← 原始记录和执行日志
```

> **重要原则**：不要跳到第四层设计第三层。**先执行→再总结→再封装**。顺序不可逆。
