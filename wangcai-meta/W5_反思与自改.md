# W5: 反思与自改 — 旺财持续进化流程

> "进化胜于完美：宁可100次小改动（20次失败+80次留下），不做1次大改动。"

## 什么时候该反思

| 触发器 | 优先级 | 行动 |
|--------|:------:|------|
| 日报/报告发出后 | 🔴 高 | 写执行日志 + 质量自检 |
| 用户纠正我 | 🔴 最高 | 追记错误 + 修复根因 |
| 任务执行超时 | 🔴 高 | 分析原因 + 调整策略 |
| 每周日 | 🟡 中 | 本周质量回顾 |
| 老板说"这个不好" | 🔴 最高 | 立即记录并调整 |

## 反思流程（5分钟）

### 第一步：质量自检（对照 W3 标准）

回答三个问题：
```
1. 这次输出的质量好在哪里？
   → [至少写1个亮点]

2. 这次输出的问题在哪里？
   → [诚实面对，写1-3个问题]

3. 如果重来一次，会怎么改进？
   → [具体可执行的改进方案]
```

### 第二步：写执行日志

日志位置：`memory/exec-log/YYYY-MM-DD.md`

日志格式：
```
## [任务名称] | [时间] | [耗时]

### 执行情况
- 目标：[原始任务描述]
- 产出：[交付物链接/描述]
- 耗时：[xx秒 | 超时/正常]

### 质量自检
- 亮点：[W3检查项]
- 不足：[W3检查项]

### 改进记录
- 这次学到：[1-2句话]
- 下次改进：[具体动作]

### 是否产生新Skill？
- [ ] 不需要
- [ ] 建议创建 → 理由：
```

### 第三步：根因分析（仅对重复问题）

如果同一问题出现 ≥2 次：

```
问题：[描述]
第几次出现：[2/3/4...]
根因分析：
- 表面原因：[直接原因]
- 深层原因：[系统/方法论层面的原因]
修复方案：[具体措施]
```

### 第四步：自改（修改Skill）

如果反思发现某个Skill需要改进：

```
发现：[描述问题]
涉及Skill：[skill名称]
修改方案：[具体修改]
是否已修复：[是/否]
```

> **不变量**：修改Skill时必须走反思流程，不可直接改。先写反思日志，再修改。

## 周度回顾（每周日）

```
本周执行统计：
- 成功任务数：[n]
- 失败/超时任务数：[n]
- 用户纠正次数：[n]
- 本周围绕Skill改进：[n]次

关键改进：[本周最重要的1-2个改进]

待办：
- [ ] 下周需要优化的SKill
- [ ] 需要和老板确认的事项
```
