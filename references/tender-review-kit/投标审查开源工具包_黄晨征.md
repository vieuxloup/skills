## 摘要

该项目（tender-review-kit）将投标审查从人工翻阅标书升级为可复现的工程流程。核心思路是**三层分工**：确定性能完成的交给程序，需要语义判断的交由大模型，最后用反向校验堵住遗漏。

- **GitHub**: [github.com/matongAI-lab/tender-review-kit](https://github.com/matongAI-lab/tender-review-kit)
- **许可证**: MIT

## 一、痛点

漏一个签章、漏一个承诺函、漏一个「必须提供原厂授权」、漏一个「▲ 参数必须响应」，可能不是扣几分，而是直接**无效投标**。

## 二、三层分工架构

| 层级 | 职责 | 工具 |
|:---|:---|:---|
| Layer 1 | 判词撒网 — 关键词库逐行扫描，标记风险位置 | `data/keywords.json`（108个基础判词） |
| Layer 2 | 语义判断 — 大模型结合上下文判断废标/普通约束 | 大模型（Claude/GPT） |
| Layer 3 | 程序反查 — 反向检查判词命中是否被清单覆盖 | `check_coverage.py` |

**核心原则**: 大模型不负责「有没有漏」——程序自己不能审计自己。

## 三、产出：五张可执行清单

1. **废标/否决风险清单** — 每条带原文出处和核对动作
2. **评分项清单** — 价格分、商务分、技术分、评分梯度
3. **证明材料清册** — 资格文件、资质证书、业绩证明等横向聚合
4. **▲ / ★ 标识参数清单** — 区分废标风险与普通响应要求
5. **时间节点清单** — 投标截止、开标、答疑、质疑等

## 四、两层防线

- **防漏抄**: `check_coverage.py` 反向检查判词命中是否被清单覆盖
- **防判断盲区**: A/B 红蓝对抗 — 两个独立大模型流程分别审同一份文件，对比差异

## 五、快速上手

```bash
提取文本 → 判词扫描 → 大模型整理工作区 → 程序护栏反查 → 导出 Excel
```

## 六、关键设计理念

1. **判词库会越用越厚** — 知识沉淀进 `data` 和 `references`，而非散落在一次性对话中
2. **不是黑盒按钮** — 每条风险可追溯到原文行号，可复核、可修改、可沉淀
3. **社区版定位**: 可复现的审标底盘，不是全自动投标审查保险箱
4. **不适合的情况**: 行业强专业标书（医疗、军工、科研）、大量扫描图、资格条件绕、标书前后矛盾
