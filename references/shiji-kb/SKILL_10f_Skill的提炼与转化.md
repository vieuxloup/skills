# SKILL 10f: Skill的提炼与转化

> **核心理念**：Skill不是Spec，而是可执行的工程规范。简练、可测、可维护。

---

## 一、问题诊断

### 1.1 当前问题

❌ **过于冗长**：部分Skill像技术文档，而非操作手册  
❌ **缺乏工程化**：Skill与Script脱节，缺乏验证机制  
❌ **维护不足**：缺少定期审查机制和质量度量标准

### 1.2 理想状态

✅ **简练**：核心内容300-500行，开门见山  
✅ **工程化**：每个Skill关联具体脚本/工具，提供检查清单  
✅ **可维护**：定期Lint检查，版本化管理，及时清理过时内容

## 二、Skill编写规范

### 2.1 标准结构

详见：[SKILL_10f1_模板库](./references/SKILL_10f1_模板库.md)

**核心章节**（必需）：
1. **快速开始**：何时使用、核心步骤（3-5步）、成功标准
2. **工具与脚本**：关联脚本列表、使用示例
3. **检查清单**：执行前/中/后验证

**可选章节**：
- 详细说明（关键概念、注意事项）
- FAQ
- 附录（关联文档）

### 2.2 三级命名规范

**设计理念**：扁平化目录结构，便于人类理解，无需翻阅多层文件夹。

#### 命名层级

```
skills/
├── SKILL_03_实体构建.md              # 第一级：工作阶段总览
├── SKILL_03a_实体标注.md             # 第二级：具体工作步骤
├── SKILL_03b_人名消歧.md             # 第二级：具体工作步骤
└── references/
    ├── SKILL_03a1_标注规则库.md      # 第三级：步骤的具体规则/数据
    └── SKILL_03a2_实体类型定义.md    # 第三级：步骤的具体规则/数据
```

**层级说明**：

| 层级 | 命名格式 | 位置 | 用途 | 示例 |
|------|---------|------|------|------|
| **一级** | `SKILL_XX_工作名` | `skills/` | 阶段总览，串联流程 | `SKILL_03_实体构建.md` |
| **二级** | `SKILL_XXa_步骤名` | `skills/` | 具体操作步骤，可独立执行 | `SKILL_03a_实体标注.md` |
| **三级** | `SKILL_XXa1_规则名` | `skills/references/` | 规则库、词表、配置数据 | `SKILL_03a1_标注规则库.md` |

#### 为什么不用Anthropic官方结构

✅ **扁平化优势**：
- 人类阅读更友好：无需在多层目录中跳转
- 文件名即层级：`SKILL_03a` 清楚表示属于 `SKILL_03`
- 引用路径简洁：`[SKILL_03a](SKILL_03a_实体标注.md)`
- IDE搜索高效：直接搜索 `SKILL_03` 即可找到所有相关文件

❌ **嵌套结构劣势**：
- 需要频繁切换目录
- 文件名缺乏上下文（多个 `skill.md` 难以区分）
- 引用路径冗长

**详细拆分策略**：参见 [SKILL_10f2_精简拆分案例库](./references/SKILL_10f2_精简拆分案例库.md)

### 2.3 长度控制

| 内容类型 | 推荐长度 | 最大长度 |
|---------|---------|---------|
| 快速开始 | 30-50行 | 100行 |
| 详细说明 | 50-100行 | 200行 |
| 工具与脚本 | 20-30行 | 50行 |
| 检查清单 | 10-20行 | 30行 |
| FAQ | 20-50行 | 100行 |
| **总计** | **200-300行** | **500行** |

**超过500行的Skill**：
- 考虑拆分为多个子Skill（参见[2.2节三级命名规范](#22-三级命名规范)）
- 将详细案例移至 `labs/examples/`
- 将背景理论移至 `docs/theory/`

### 2.4 写作原则

#### 原则1：开门见山

✅ 直接给出操作步骤  
❌ 不要以大量背景介绍开头

#### 原则2：可执行优先

每个步骤必须是**可直接执行的动作**：

| 类型 | 示例 |
|-----|------|
| ✅ 可执行 | "运行 `python scripts/lint.py`" |
| ✅ 可执行 | "打开文件 `chapter_md/001.md`" |
| ❌ 不可执行 | "确保数据质量" |
| ❌ 不可执行 | "理解本体结构" |

#### 原则3：检查清单必需

每个Skill必须包含：
- **执行前检查**：环境、依赖、前置条件
- **执行中验证**：关键步骤完成标志
- **执行后验证**：输出质量、成功标准

## 三、Skill工程化

### 3.1 Skill与脚本关联

每个Skill应关联具体的脚本/工具，形成闭环。

**示例**：
```
SKILL_01_古籍校勘.md
    ├─ scripts/validation/lint_text_integrity.py
    ├─ scripts/curation/compare_versions.py
    └─ scripts/curation/generate_collation_report.py
```

### 3.2 脚本组织规范

按功能分组：

- `scripts/validation/` - 校验脚本
- `scripts/generation/` - 生成脚本
- `scripts/conversion/` - 转换脚本
- `scripts/reflection/` - 反思脚本
- `scripts/workflows/` - 工作流编排

**详细示例**：参见 [SKILL_10f4_脚本分解示例](./references/SKILL_10f4_脚本分解示例.md)

## 四、Skill质量标准

### 4.1 基本要求

- [ ] 包含YAML frontmatter（name, title, description）
- [ ] 包含"快速开始"章节
- [ ] 包含"工具与脚本"章节（至少1个关联脚本）
- [ ] 包含"检查清单"章节
- [ ] 总长度<500行
- [ ] 所有脚本路径有效
- [ ] 所有Skill引用有效

### 4.2 质量度量

| 指标 | 标准 | 检查方法 |
|-----|------|---------|
| 长度 | <500行 | `wc -l skills/SKILL_XX.md` |
| 引用频率 | ≥3次/月 | 搜索日志/代码引用 |
| 脚本关联 | ≥1个 | `grep "scripts/" SKILL_XX.md` |
| 链接有效性 | 100% | 运行link checker |
| 更新频率 | 每季度审查 | 检查最后修改日期 |

## 五、从Spec到Skill的转化流程

### 5.1 转化标准

| Spec（技术文档） | Skill（操作规范） |
|----------------|----------------|
| 描述"是什么" | 指导"怎么做" |
| 理论为主 | 实践为主 |
| 面向研究者 | 面向Agent+开发者 |
| 长篇幅（>1000行） | 精简（<500行） |

### 5.2 转化步骤

1. **提取核心流程**：识别3-5个关键步骤
2. **识别关联工具**：列出所有相关脚本
3. **设计检查清单**：定义执行前/中/后验证点
4. **精简背景说明**：移至 `references/` 或删除
5. **添加实例**：1-2个典型使用场景

### 5.3 从项目执行到 Skill（姊妹规则）

做完一次具体项目后，常有把"经验"沉淀成 Skill 的冲动。**不要新建一个以项目命名的 Skill**，而应该把产物**拆分**为两类：

| 类型 | 归属 | 判断标准 |
|-----|------|---------|
| **可跨项目复用的规律** | 升入相关领域 Skill（如 `SKILL_01i`、`SKILL_03a`） | 下次启动同类任务时会查阅 |
| **项目特定的复盘叙事** | 归入 `doc/curation/`（如 `doc/curation/reports/<项目名>_整理复盘.md`） | 只对这次项目有效，时间线/踩坑/未完成事项 |

**反例**（一个真实的踩坑）：曾把《读史记十表》OCR 整理经验做成 `SKILL_10i_OCR古籍整理项目经验`——结果发现其内容是两类混合，且与 10f 自身边界模糊。正确做法是：
- 方法论（OCR 错字归纳、引号作用域栈、LLM 偏差前置提示）→ `SKILL_01i` 对应章节
- 项目叙事（这次做了什么、踩了什么坑）→ `doc/curation/reports/读史记十表_整理复盘.md`

**判断口诀**：*"某一本书 / 某一次事件" 专属的内容不进 Skill。*

## 六、Skill精简与拆分

**何时需要精简**：

- [ ] Skill文件超过500行
- [ ] 代码示例超过200行
- [ ] 详细配置说明超过100行
- [ ] 用户反馈"太长，找不到重点"

**拆分流程**：

1. 备份原文件
2. 按三级命名创建references文件
3. 精简主文档（删除详细代码/案例/FAQ，保留引用）
4. 更新链接
5. 验证完整性

**详细方法**：参见 [SKILL_10f2_精简拆分案例库](./references/SKILL_10f2_精简拆分案例库.md)

## 七、Skill的更新与维护

### 7.1 更新触发条件

- [ ] 数据规模变化（>10%）
- [ ] 文件路径变更
- [ ] 脚本逻辑更新
- [ ] 工具链升级
- [ ] 过时内容

### 7.2 标准更新流程

1. 触发更新识别
2. 全面检查（统计数据、文件路径、脚本逻辑、索引文件）
3. 批量更新
4. 验证完整性

**详细流程**：参见 [SKILL_10f3_更新维护指南](./references/SKILL_10f3_更新维护指南.md)

### 7.3 定期维护

- **月度检查**：运行Skill新鲜度检查，检查失效链接
- **季度审查**：识别过时Skill，更新所有统计数据，重构超长Skill

## 九、FAQ

### Q1: 什么时候需要创建新Skill？

**A**: 满足以下任一条件：
- 有完整的工作流程（3步以上）
- 需要多次重复执行
- 有明确的输入/输出
- 涉及多个工具/脚本

### Q2: Skill与脚本的关系是什么？

**A**:
- **Skill是流程规范**：定义"做什么、怎么做、检查什么"
- **脚本是工具**：自动化Skill中的可编程步骤
- **关系**：Skill引用脚本，脚本支撑Skill

### Q3: 如何判断Skill是否需要重构？

**A**: 出现以下信号时考虑重构：
- 长度超过500行
- 最近30天引用次数<3
- 关联脚本失效
- 内容与其他Skill重复>50%
- 有用户反馈"难以理解"

### Q4: Skill与文档的区别？

**A**:
| 类型 | 目标读者 | 内容 | 位置 |
|-----|---------|------|------|
| Skill | AI Agent + 开发者 | 可执行的操作规范 | `skills/` |
| 用户文档 | 终端用户 | 功能说明、使用指南 | `docs/` |
| 理论文档 | 研究者 | 背景、设计思路 | `docs/theory/` |
| 工作日志 | 项目组内部 | 进度、问题记录 | `logs/` |
| **项目复盘** | 项目组内部 | 某次执行的时间线、踩坑、未完成事项 | `logs/` |

### Q5: 项目复盘（postmortem / retrospective）应该作为 Skill 吗？

**A**：**不应该**。项目复盘属于 `logs/`，不属于 `skills/`。

- **Skill** 是可跨项目复用的操作规范——下次启动同类任务时会查阅
- **项目复盘** 是某次具体执行的叙事——时间线、踩的坑、未完成事项，只对这次项目有效

做完一个项目后，若有沉淀意愿，按 §5.3 的姊妹规则拆分：可复用规律升入相关 Skill，项目叙事入 doc/curation。案例：《读史记十表》OCR 整理——方法论入 [SKILL_01i](./SKILL_01i_OCR影印古籍句读排版.md)，项目叙事入 `doc/curation/reports/读史记十表_整理复盘.md`。

**判断口诀**：*"某一本书 / 某一次事件" 专属的内容不进 Skill。*
