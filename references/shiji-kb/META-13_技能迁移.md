# 元技能13: 技能迁移 — SKILL跨项目复用方法论


---

## 一、核心思想

> **"SKILL文档是可迁移的知识资产"**

**技能迁移**（Skill Transfer）是将在一个项目/数据集中积累的SKILL文档、工具、方法论迁移到新的领域/数据集的过程。

### 应用场景

**现状**:完成了《史记》130篇的知识抽取

**未来目标**:
- 《汉书》(120篇) — 同为纪传体,结构相似
- 《资治通鉴》(294卷) — 编年体,结构不同
- 《二十四史》— 规模扩大10倍+

**挑战**:
- 相似但不完全相同(如《汉书》多"表"少"世家")
- 文体差异(纪传体 vs 编年体)
- 规模差异(130篇 vs 294卷 vs 3000+卷)

---

## 二、迁移层次

### 2.1 直接复用（Level 1）

**适用场景**:目标数据与源数据高度相似

**可直接复用的内容**:
- **标注体系**(18类Token):`〖@人名〗`、`〖=地名〗`等
- **Lint工具**:`lint_text_integrity.py`、`lint_markdown.py`
- **通用SKILL**:00-META系列(反思、质量控制、可读性等)
- **基础设施**:紫色编号、双格式输出、Git工作流

**示例**:《史记》→《汉书》

**操作逻辑**:
1. 复制SKILL文档到新项目
2. 复制Lint工具
3. 复制标注规范
4. 开始标注试点(冷启动)
   - 预期: 80%规则直接可用,20%需调整

---

### 2.2 适配调整（Level 2）

**适用场景**:有部分差异,需调整参数/规则

**需要调整的内容**:

#### 差异1:结构变化

```markdown
《史记》结构:
- 12本纪、10表、8书、30世家、70列传

《汉书》结构:
- 12纪、8表、10志、70传

《资治通鉴》结构:
- 16朝纪、294卷(无表/书/传区分)
```

**调整逻辑**:
- 需修改章节类型映射配置
- 《史记》: 本纪/表/书/世家/列传
- 《汉书》: 纪/表/志/传（结构简化）

---

#### 差异2:实体类型分布变化

```markdown
《史记》特点:
- 战国至汉初,邦国众多('邦国标记密集)
- 神话传说多(黄帝/尧舜,神话标记必要)

《汉书》特点:
- 西汉一朝,邦国减少(主要是汉朝)
- 官职体系更复杂(需扩展官职词表)

《资治通鉴》特点:
- 跨16朝,年号复杂(需专门的年号实体类型)
- 编年体,时间标注更密集
```

**调整策略**:
- 针对《汉书》:扩展官职词表（史记230个 + 汉代新增120个）
- 针对《资治通鉴》:新增年号标注类型（18类 → 19类）

---

#### 差异3:事件类型分布

```markdown
《史记》事件分布:
- 战争事件: 23% (战国纷争)
- 继位事件: 18% (朝代更迭频繁)

《汉书》事件分布(预测):
- 战争事件: 15% (一朝之内,战争减少)
- 政治改革: 25% (汉朝制度建设)
```

**调整逻辑**:
- 事件类型权重需根据时代背景调整（影响LLM prompt）
- 《史记》: 战争23% / 继位18% / 政治15%
- 《汉书》: 战争15%↓ / 政治改革25%↑ / 继位10%↓

---

### 2.3 重新设计（Level 3）

**适用场景**:结构根本不同,需重新设计

**需要重新设计的内容**:

#### 差异1:编年体 vs 纪传体

```markdown
《史记》(纪传体):
- 以人物/主题为线索
- 事件分散在多篇中
- 需要跨章对齐(cross_ref关系)

《资治通鉴》(编年体):
- 以年份为线索
- 同一年的事件集中记述
- 不需要跨章对齐(已按年排列)
```

**重新设计逻辑**:
- 《史记》工序: 事件构建 → 事件关系发现(跨章对齐)
- 《资治通鉴》工序: 事件构建 → 编年排序(时间对齐)
  - 不需要cross_ref关系(已按年排列)
  - 但需要"同年多事件"的并发关系建模

---

#### 差异2:多源互见 vs 单一叙述

```markdown
《史记》:
- 同一事件多章记述(本纪/世家/列传)
- 需要多源融合

《汉书》:
- 类似史记,也有多源记述

《资治通鉴》:
- 司马光已做融合(综合多史书)
- 单一叙述线,无需再融合
```

**设计调整逻辑**:
- 《史记》/《汉书》: 需要多源事件融合SKILL
- 《资治通鉴》: 不需要多源融合，但需要史源追溯（记录引用了哪些史书）

## 三、迁移方法论

### 3.0 核心原则：反思与迭代贯穿全程

**关键洞察**：迁移不是一次性任务，而是"粗标→观察→反思→精标→验证"的螺旋上升过程

#### 在每个工序中应用反思和迭代

| 工序 | 机器的角色 | 人工的角色 | 反思内容 | 迭代目标 |
|------|-----------|-----------|---------|---------|
| **实体标注** | 粗标试点章节 | 对比原文观察遗漏/误标 | 是否需要新增实体类型？ | 遗漏率↓50%，误标率↓30% |
| **结构分析** | 提取章节结构 | 抽查结构准确性 | 新文体结构是否识别？ | 结构识别准确率>95% |
| **事件抽取** | 提取事件三元组 | 抽查事件完整性 | 事件类型分布是否合理？ | 事件召回率>80% |
| **关系发现** | 发现实体关系 | 验证关系准确性 | 是否有新关系类型？ | 关系准确率>85% |
| **质量控制** | 运行Lint检查 | 人工审查错误报告 | 哪些规则需要调整？ | 错误率<3% |

**人工在迭代中的三大职责**:
1. **观察**：对比原文 vs 标注结果，发现系统性遗漏和误标
2. **抽样质检**：每章抽查100-200个样本，统计错误率
3. **决策**：基于Agent反思建议，决定采纳/调整/拒绝

**迭代收敛标准**:
- 第1轮：快速粗标，错误率可达30-50%（可接受）
- 第2轮：反思后精标，错误率应降至10-15%
- 第3轮：细化调整，错误率应降至3-5%
- 第N轮：持续迭代直至错误率<3%

---

### 3.1 试点驱动的迁移（Pilot-driven Transfer）

**核心**:不要一次性迁移全部,用小规模试点发现问题

**重点关注**：实体类型是迁移第一关

> **为什么实体类型最重要？**
>
> 不同的书籍，实体类型分布差异极大：
> - 《史记》：邦国密集（战国七雄）、神话人物（黄帝/尧舜）
> - 《汉书》：年号密集（建元/元狩）、三公九卿（大司马/大司徒）
> - 《资治通鉴》：年号更复杂（跨16朝）、史源引用（"据《XX》"）
>
> **实体类型错误会传播到后续工序**：
> - 实体标注遗漏 → 事件抽取不完整 → 关系发现缺失 → 知识图谱质量低
>
> **因此必须在第一阶段就解决实体类型适配问题**

#### 阶段1:选择试点章节（3-5篇）

**选择原则**:
- **覆盖多样性**:选择不同类型的章节
  - 《汉书》试点:1篇纪、1篇表、1篇志、2篇传
- **难度分层**:简单→中等→复杂
  - 简单:人物少、事件少
  - 复杂:人物多、跨章引用多

**《汉书》试点候选**:
```markdown
1. 《高帝纪上》— 与《史记·高祖本纪》高度重叠,便于对比
2. 《百官公卿表》— 表类文献,测试结构化提取
3. 《礼乐志》— 志类文献,测试新文体
4. 《韩信传》— 与史记对比,测试差异
5. 《儒林传》— 测试人物关系网络
```

---

#### 阶段2:粗标试点（机器优先，快速迭代）

**核心原则**："先粗标，观察差异，再精细化"

**操作流程**:
1. **准备阶段**
   - 复制SKILL文档到新项目
   - 复制标注规范（保持18类Token不变）
   - 准备3-5个试点章节

2. **第一轮粗标**（机器为主）
   - 使用现有SKILL让LLM对试点章节进行自动标注
   - **不追求完美**，目标是快速生成初版标注
   - 运行Lint工具，记录格式错误

3. **对比观察**（人工关键步骤）
   - **并排对比**：原文 vs 标注文本（去除标记符号后）
   - **检查遗漏**：哪些实体未被标注？（如新的官职、地名、制度名）
   - **检查误标**：哪些标注是错误的？（类型错误、边界错误）
   - **记录模式**：遗漏/误标是否有规律？（如某类词汇系统性遗漏）

4. **抽样质检**（人工验证）
   - 每章抽查100-200个标注实体
   - 统计遗漏率和误标率
   - 识别高频错误模式（如"某类官职全部遗漏"）

**输出**：
- 初版标注文件（含错误）
- 遗漏实体清单（人工观察记录）
- 误标案例清单（人工观察记录）
- 错误模式总结

---

#### 阶段3:反思实体类型差异（Agent + 人工）

**核心任务**：基于粗标结果，反思是否需要新增/修改实体类型

**反思提示词**（给LLM）:
```markdown
# 任务：分析《汉书》实体类型需求

## 输入材料
1. 《史记》标注体系：18类Token
2. 《汉书》试点章节原文（5篇）
3. 第一轮粗标的遗漏实体清单（人工观察）
4. 第一轮粗标的误标案例清单（人工观察）

## 反思问题
1. **遗漏分析**：哪些遗漏的实体无法用现有18类标注？
   - 示例：年号"建元元年"，现有类型〖%时间〗是否足够？
2. **类型冲突**：是否有实体在《汉书》中类型不同？
   - 示例："大司马"在史记可能是人名上下文，在汉书是专门官职
3. **新增建议**：是否需要新增实体类型？
   - 提供3-5个候选，说明理由和影响范围
4. **合并建议**：是否有类型可以合并？
   - 如〖^官职〗和〖◆谥号〗是否可以合并为〖^职衔〗

## 输出格式
- 保持现有类型（列出理由）
- 新增类型（名称/符号/定义/示例/影响章节数）
- 修改类型（原定义→新定义，理由）
- 废弃类型（理由）
```

**人工审查**（关键环节）:
- 检查Agent建议是否合理（是否理解了汉书特点）
- 评估新增类型的成本（标注工作量 vs 收益）
- 决策：采纳/调整/拒绝建议

**输出**：
- 标注体系v2.0规范（如18类→19类，或保持18类但扩展定义）
- 新增类型的标注指南

#### 阶段4:第二轮标注（应用反思结果）

**操作流程**:
1. 更新SKILL文档（加入汉书特有场景）
2. 扩展词表（补充汉代官职/地名等）
3. 重新标注试点章节（使用v2.0规范）
4. **对比v1 vs v2**：
   - 遗漏率是否降低？（目标：降低50%+）
   - 误标率是否降低？（目标：降低30%+）
5. 人工抽样质检（每章100个实体）

**如果质量未达标** → 回到阶段3（第二轮反思）

#### 阶段5:记录所有问题（完整错误日志）

**关键**：记录第二轮迭代后仍存在的问题

创建文档: `doc/transfer/史记→汉书_迁移日志.md`

```markdown
# 《史记》SKILL迁移到《汉书》错误日志

## 试点章节:高帝纪上

### 错误1:官职标注不完整
**现象**: 《汉书》出现"大司马"、"大司徒"等三公官职,但《史记》官职词表中没有
**原因**: 汉武帝改制后新设三公九卿体系
**影响章节**: 高帝纪上、百官公卿表
**修正方案**: 扩展官职词表,新增30个汉代官职
**状态**: 待修正

### 错误2:"表"的格式无法解析
**现象**: `extract_events.py`无法处理《百官公卿表》的表格结构
**原因**: 《史记》的"表"主要是年表(时间线),《汉书》的"表"是职官表(矩阵)
**影响章节**: 百官公卿表、古今人表
**修正方案**: 新增`extract_table_structured.py`处理二维表格
**状态**: 待开发

### 错误3:年号标注缺失
**现象**: "建元元年"、"元狩五年"等年号未标注
**原因**: 《史记》时代年号制度未完全确立,《汉书》大量使用年号纪年
**影响章节**: 所有纪、传
**修正方案**:
  - 新增`〖*年号〗`标注类型
  - 更新`SKILL_02_结构分析.md`,增加年号章节
  - 开发`parse_reign_periods.py`解析年号
**状态**: 待讨论(是否新增类型 vs 复用`〖%时间〗`)

### 错误4:交叉引用模式变化
**现象**: 《汉书》引用《史记》(如"见《史记》"),这种跨书引用无法用cross_ref关系表示
**原因**: 《史记》的cross_ref是书内跨章,《汉书》出现跨书引用
**影响章节**: 多处(约20+引用)
**修正方案**: 新增关系类型`external_ref`
**状态**: 待开发

... (继续记录20-30个问题)
```

#### 阶段6:统计错误分布

**分析维度**:

**输出示例**:

- 新文体结构: 5个问题 (25%)
- 标注类型缺失: 4个问题 (20%)
- 关系类型扩展: 3个问题 (15%)
- 影响章节数: 5/5 (100%)
- 预估修正工作量: 45小时

### 3.2 反思驱动的适配（Reflection-driven Adaptation）

**流程**: 错误日志 → Agent反思 → SKILL修正

#### 步骤1:生成反思提示词

**提示词结构**:
- 背景：说明迁移项目和发现的问题数量
- 原始SKILL：读取源SKILL文档内容
- 目标语料特点：描述目标语料的结构/时代/文体差异
- 错误日志：完整错误列表
- 反思任务：
  1. 根因分析（为什么SKILL规则失效）
  2. 共性模式（是否有系统性问题）
  3. 修正方案（泛化/新增/废弃规则）
  4. 优先级排序（按影响范围）
- 输出格式：Markdown格式（根因分析/共性模式/修正建议/新增SKILL草稿）

---

#### 步骤2:Agent反思

**操作逻辑**:
- 调用LLM API(Claude/GPT)进行反思
- 输入: 错误日志 + 原SKILL文档 + 目标语料特点
- 输出: 反思报告（根因分析/共性模式/修正建议）

**反思输出示例** (`doc/transfer/反思报告_实体标注.md`):

```markdown
# 《史记》→《汉书》实体标注SKILL迁移反思

## 一、根因分析

### 问题根源1:历史时代差异
《史记》覆盖黄帝至汉武帝(约2600年),官职体系多变。
《汉书》仅覆盖西汉一朝(约230年),官职体系固定为三公九卿制。

**导致**:
- 《史记》官职词表以战国/秦制为主
- 《汉书》需要西汉官制词表

### 问题根源2:文体结构差异
《史记》的"表"以年表为主(时间轴 + 事件)。
《汉书》的"表"增加了职官表、地理表(二维矩阵)。

**导致**:
- 《史记》的`extract_events.py`假设表格是时间线
- 《汉书》的表格是矩阵,需要新的解析逻辑

---

## 二、共性模式

### 模式1:词表不完整(40%问题)
**表现**:新增实体类型/词汇在原词表中不存在
**案例**:大司马、大司徒、大司空、太尉、御史大夫等汉代三公九卿
**泛化**:任何朝代迁移都会遇到此问题

### 模式2:文体假设失效(25%问题)
**表现**:代码中硬编码了《史记》的文体结构
**案例**:`CHAPTER_TYPES = {'本纪', '表', '书', '世家', '列传'}`写死
**泛化**:跨语料迁移必须参数化文体类型

### 模式3:标注粒度不足(20%问题)
**表现**:原标注体系粒度不够细,无法区分新场景
**案例**:年号"建元元年"无专门标注,复用`〖%时间〗`导致后续解析困难
**泛化**:标注体系需要版本演化机制(v2.5 → v3.0)

---

## 三、修正建议(优先级排序)

### P0:扩展词表(影响100%章节,工作量8小时)
**任务**:
1. 提取《汉书》前20篇的所有官职标注
```

## 八、迁移到非古籍与通用书籍

### 8.1 核心挑战

从古籍知识工程迁移到**现代书籍**或**通用文本**，面临的核心差异：

| 维度 | 古籍（史记/汉书） | 现代书籍/通用文本 |
|------|------------------|------------------|
| **文本结构** | 高度结构化（纪传体/编年体） | 松散结构（章节/段落） |
| **实体密度** | 极高（5-10个/100字） | 中低（0.5-2个/100字） |
| **实体类型** | 历史专属（人名/官职/邦国） | 领域相关（人物/机构/产品等） |
| **时间表达** | 复杂纪年（年号/干支） | 标准年月日 |
| **关系复杂度** | 高（家族/政治/战争） | 中（组织/引用/因果） |
| **标注挑战** | 古文语义理解 | 歧义消解/同名异指 |

### 8.2 迁移层次（通用书籍适配）

#### Level 1: 直接复用（元技能层，80%+可用）

**高度可迁移的内容**：
- ✅ **00-META-00 到 00-META-13**: 所有元技能文档
  - 反思、迭代、冷启动、柳叶刀方法等核心方法论
  - 与具体领域无关，适用于任何知识工程项目
- ✅ **可读性设计**: Token标注优于XML的原则
- ✅ **质量控制体系**: 五层Lint架构
- ✅ **Agent提示词工程**: SKILL驱动的提示词设计

#### Level 2: 适配调整（标注体系，50%可用）

**需要调整的实体类型**：

**古籍18类实体** → **通用文本实体类型**

```markdown
保留（通用性强）：
✅ @人名   → @Person（扩展：组织机构人员）
✅ =地名   → =Location（扩展：虚拟地点）
✅ %时间   → %Time（简化：标准时间格式）
✅ #数量   → #Quantity
✅ ~引文   → ~Citation

需要替换（领域特定）：
❌ ^官职   → @Role（通用角色/职位）
❌ &邦国   → @Organization（组织/机构/公司）
❌ &氏族   → @Family（家族/团体，可选）
❌ ^制度   → @Concept（抽象概念）
❌ [刑法   → @LegalTerm（法律术语，可选）
❌ '谥号   → 删除（古籍特有）

新增（现代文本需要）：
➕ @Product（产品/品牌）
➕ @Event（事件名称，如"巴黎协定"）
➕ @Technology（技术术语）
➕ $Currency（货币）
➕ @Email, @URL（结构化信息）
```

**标注体系适配示例**：

```markdown
古籍原文:
〖@秦始皇〗于〖%秦王政二十六年〗灭〖&六国〗，称〖^皇帝〗。

现代文本:
〖@埃隆·马斯克〗于〖%2022年〗收购〖@Twitter〗，担任〖@CEO〗。
```

#### Level 3: 重新设计（事件Schema，30%可用）

**古籍事件类型** vs **通用事件类型**：

```markdown
古籍事件（高度专业化）：
- 战争、政变、改革、外交、祭祀、天象...

通用事件（需重新定义）：
- 商业事件：收购/融资/上市/倒闭
- 科技事件：发布/突破/专利
- 社会事件：会议/运动/政策
- 文化事件：出版/展览/获奖
```

**事件Schema适配原则**：
1. 保留通用字段：`event_id`, `event_name`, `event_type`, `participants`, `location`, `time`
2. 删除古籍特有字段：`dynasty`, `reign_year`, `calendar_system`
3. 新增现代字段：`url`, `source_credibility`, `impact_score`

### 8.3 试点策略（通用书籍）

#### 试点选择原则

**不要选择**：
- ❌ 小说/文学作品（虚构，实体关系弱）
- ❌ 高度技术文档（术语密集，需专业知识）
- ❌ 新闻快讯（碎片化，缺乏深度关系）

**推荐试点**：
- ✅ **传记类书籍**（如《乔布斯传》）
  - 与古籍"列传"结构相似
  - 人物关系密集
  - 时间线清晰
- ✅ **历史通俗读物**（如《人类简史》）
  - 事件驱动
  - 概念丰富
  - 结构化章节
- ✅ **商业案例集**（如《创新者的窘境》）
  - 实体类型明确（公司/产品/人物）
  - 因果关系清晰

#### 试点规模

```markdown
第一阶段（MVP）:
- 1本书
- 1-3章（约5000-10000字）
- 目标：验证标注体系适配性

第二阶段（扩展）:
- 同一本书的完整标注
- 目标：积累错误模式，完善SKILL

第三阶段（泛化）:
- 3-5本不同类型的书
- 目标：提炼通用方法论
```

### 8.4 关键差异的应对策略

#### 差异1：结构化程度降低

**古籍优势**：
- 纪传体天然的人物索引
- 表的时间锚点
- 书的分类体系

**现代书籍问题**：
- 章节划分不规则
- 缺乏统一的时间线
- 内容跳跃性强

**应对策略**：
1. **人工预处理**：为每章添加`summary`和`key_entities`元数据
2. **双层标注**：先标注章节级大纲，再标注段落级实体
3. **弱化时间线**：不强求所有事件都有精确时间

#### 差异2：实体歧义性增加

**古籍相对确定性**：
- "秦始皇"在《史记》中指向唯一
- 地名有明确地理边界
- 官职有标准定义

**现代文本挑战**：
- "苹果"可能是水果/公司/品牌/手机
- "华为"可能指公司/产品系列/技术标准
- 同名人物频繁（需上下文消歧）

**应对策略**：
1. **类型前缀强制**：`〖@苹果公司〗` vs `〖#苹果〗`（数量）
2. **上下文窗口扩大**：从±100字扩展到±200字
3. **消歧提示词增强**：显式提供领域背景

#### 差异3：关系发现难度增加

**古籍关系密集**：
- 家族关系：父子/君臣关系明确
- 战争关系：攻守双方清晰
- 时序关系：纪年体系统一

**现代文本关系隐含**：
- 组织关系：隶属/合作/竞争需推理
- 引用关系：观点引用需理解语义
- 因果关系：跨段落因果链复杂

**应对策略**：
1. **分层关系发现**：
   - L1: 共现关系（自动，基于统计）
   - L2: 显式关系（LLM提取，如"X创办Y"）
   - L3: 隐式关系（人工标注+Agent推理）
2. **降低关系完整性要求**：从"完整图谱"降级为"核心关系骨架"
3. **用户反馈循环**：允许读者标注缺失关系

### 8.5 成功案例构想

## 九、总结

### 核心要点

1. **实体类型优先**：不同书籍实体类型差异极大，必须先粗标→观察→反思→确定类型体系
2. **机器+人工协作**：机器快速粗标，人工对比观察遗漏和误标，识别模式
3. **反思与迭代贯穿全程**：每个工序都要经历"粗作→观察→反思→精作→质检"的循环
4. **分层迁移**：直接复用 → 适配调整 → 重新设计
5. **试点驱动**：不要一次性迁移，用3-5篇试点章节发现问题
6. **增量验证**：每修正一个问题，立即验证效果（对比v1 vs v2）
7. **Human-in-the-loop**：关键决策需要人工审查（观察/抽样/决策），不盲目信任Agent
8. **领域扩展**：元技能高度可迁移，标注体系需适配，事件Schema需重设计

### 实战流程总结

**标准迁移流程（以实体标注为例）**：

```
第1轮（快速粗标，1-2天）
  ├─ 机器：用现有SKILL粗标试点章节
  ├─ 人工：对比原文，记录遗漏和误标（100-200个样本/章）
  ├─ 输出：遗漏清单 + 误标清单 + 错误模式
  └─ 错误率：30-50%（可接受）

第2轮（反思优化，3-5天）
  ├─ Agent：分析遗漏/误标，建议新增/修改实体类型
  ├─ 人工：审查建议，决策是否采纳
  ├─ 机器：应用v2.0规范重新标注
  ├─ 人工：抽样质检，统计遗漏率和误标率
  └─ 错误率：10-15%（需继续迭代）

第3轮（细化调整，2-3天）
  ├─ 针对高频错误调整SKILL和词表
  ├─ 重新标注，人工抽查
  └─ 错误率：3-5%（接近收敛）

第N轮（持续迭代至收敛）
  └─ 错误率<3%，通过验收

总计：约2-3周（3-5轮迭代）
```

**关键成功因素**：
- ✅ 不追求第一轮完美，快速粗标暴露问题
- ✅ 人工观察和抽样是发现模式的关键
- ✅ Agent反思提供建议，人工做最终决策
- ✅ 每轮迭代都要对比前后差异（量化改进）
- ✅ 设定明确的收敛标准（错误率阈值）

### 迁移哲学

**传统软件迁移**:
- 一次性大规模迁移
- 依赖完整的需求分析
- 失败风险高(all-or-nothing)

**本项目的迁移理念**:
- **小步快跑**:试点 → 反思 → 修正 → 扩展
- **拥抱失败**:将失败案例视为学习机会
- **知识沉淀**:每次迁移都积累经验,形成可复用资产

**核心洞察**:
> **SKILL文档不是一次性设计,而是在多项目迁移中持续演化的知识资产**

迁移过程本身就是知识提炼的过程:通过发现不同语料间的差异,我们更深刻理解了"什么是可泛化的知识工程方法论"。

### 与其他元技能的关系

- **← 冷启动**:迁移=加速的冷启动
- **← 反思**:迁移的核心驱动力
- **← 质量控制**:迁移后需验证质量不倒退
- **← 柳叶刀方法**:分层迁移对应分层分解
- **→ 规模化**:成功迁移是规模化的前提

**技能迁移是项目可持续发展的关键**:从一个史书扩展到二十四史,需要高效的迁移方法论。
